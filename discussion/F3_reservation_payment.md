# [F3] 예약 슬롯-결제 순서 방안 결정: 슬롯 우선(A) vs 결제 우선(B)

## 배경

TravelX의 결제 모델이 **전체 금액 선결제(in-app) + 지점 현금 수령** 구조로 확정되면서,
예약 플로우에서 "슬롯 배정"과 "결제"의 순서를 어떻게 가져갈지 결정이 필요합니다.

기존 cash-only 가정 하에서는 큰 이슈가 아니었지만, 앱 내 전체 금액이 실제로 PG를 통해
움직이는 구조가 되면서 순서에 따라 환불 리스크, 환율 확정 시점, 재고 정합성이 크게 달라집니다.

이 문서는 두 방안(A/B)을 비교하고, A안을 채택하는 근거와 A안 기준 구현 플로우를 정리합니다.

---

## 방안 비교

### A안: 슬롯 먼저 배정 → 결제

사용자가 지점/시간/통화를 선택하면 슬롯을 **임시 홀드**하고, 그 시점의 환율을 스냅샷으로
고정합니다. 정해진 시간(예: 5~10분) 안에 결제를 완료해야 예약이 확정되고, 시간 내 결제하지
않으면 슬롯은 자동 해제됩니다.

**특징**
- 콘서트/항공권 예매와 동일한 사용자 멘탈 모델 → 학습 비용 낮음
- 슬롯 경쟁(동시 선택 충돌)을 결제 이전, 선택 시점에 바로 걸러냄
- 환율 lock을 슬롯 홀드 시점에 명확히 표시 가능 ("이 환율로 O분 안에 결제 완료 시 확정")
- 결제 실패/타임아웃 시 되돌릴 금전 거래가 없음 (환불 프로세스 불필요)

**단점**
- 슬롯을 홀드해둔 시간 동안 다른 사용자는 해당 슬롯을 볼 수 없음 (재고 유휴 시간 발생)
- 홀드-타임아웃-재선택 UI/로직이 추가로 필요


### B안: 결제 먼저 → 슬롯 배정

사용자가 슬롯을 선택하는 즉시 결제를 진행하고, 결제 완료가 확인된 시점에 슬롯을 배정합니다.
결제 전에는 슬롯이 임시 홀드조차 되지 않습니다.

**특징**
- 슬롯 유휴 시간이 없음 (결제 완료 전까지는 다른 사용자도 동일 슬롯에 결제 시도 가능)

**단점**
- **전체 금액 결제 완료 후 슬롯이 이미 마감된 경우, 전액 환불 처리가 발생**
  - PG 수수료는 이미 발생, 정산 로직 복잡
  - 사용자 입장에서 "돈은 냈는데 자리가 없다"는 신뢰 손상 경험
- 환율 확정 시점이 애매함: 결제 시점 환율로 과금했는데 슬롯 배정이 실패하면,
  재선택 시 환율이 달라져 있을 수 있음 → 환불 후 재결제라는 이중 부담
- 동시 결제 경쟁 시 백엔드가 "결제 성공 + 슬롯 배정 실패"라는 예외 케이스를
  상시적으로 처리해야 함 (구조적으로 실패 케이스가 정상 플로우에 내장됨)

---

## 결론: A안 채택

**슬롯 우선(A) 방식으로 진행합니다.**

전체 금액 선결제 구조에서는 B안의 단점(전액 환불 리스크)이 예약금 모델보다 훨씬 크게
작용합니다. 결제된 금액이 크고, PG를 통한 실제 자금 이동이 발생하기 때문에, "결제 후
배정 실패 → 환불"이 상시 케이스로 존재하는 구조는 운영 부담과 사용자 신뢰 리스크를
동시에 키웁니다.

A안은 슬롯 배정 실패를 **결제 이전 단계에서 원천 차단**하기 때문에, 환불 프로세스 자체가
필요 없는 구조를 만듭니다. 재고 유휴 시간이라는 단점은 홀드 타이머를 짧게(5~10분)
가져가는 것으로 최소화합니다.

## A안 기준 구현 플로우

### 상태 머신

```
HELD → PENDING_PAYMENT → CONFIRMED → COMPLETED
  ↓                ↓
EXPIRED       PAYMENT_FAILED
```

| 상태 | 설명 |
|---|---|
| HELD | 슬롯 임시 잠금 + 환율 스냅샷 완료, 타이머 시작 |
| PENDING_PAYMENT | PG 결제 세션 생성됨, 사용자 결제 인증 중 |
| CONFIRMED | PG 웹훅으로 결제 성공 확인, 슬롯 영구 확정 + QR 발급 |
| EXPIRED | 타이머 만료, 슬롯 자동 해제 (스케줄러 처리) |
| PAYMENT_FAILED | 결제 실패 웹훅 수신, 슬롯 즉시 해제 |
| COMPLETED | 지점 방문 후 QR 스캔 완료, 현금 수령 완료 |

### 단계별 플로우

1. **슬롯/통화 선택 (RSV-01, 프론트)**
   사용자가 지점, 시간, 통화, 금액을 입력.

2. **슬롯 홀드 생성 (백엔드)**
   재고를 원자적으로 잠그고(동시 요청 중 하나만 성공), 현재 환율을 스냅샷으로 저장.
   상태를 `HELD`로 설정, 만료 시각(now + 5~10분) 지정.

3. **결제 세션 생성 (백엔드 ↔ PG)**
   홀드된 예약 ID와 잠긴 환율 기준 금액으로 PG 결제 세션 생성.
   상태를 `PENDING_PAYMENT`로 전환.

4. **결제 처리 (RSV-02, PG)**
   사용자가 결제 수단 인증. 프론트는 홀드 타이머 카운트다운을 계속 노출.

5. **웹훅 수신 및 확정 (백엔드)**
   PG의 결제 결과 웹훅 수신 시:
   - 예약이 아직 `HELD`/`PENDING_PAYMENT` 상태이고 만료 전이면 → `CONFIRMED` 전환, 슬롯 영구 확정, QR 발급
   - 이미 `EXPIRED` 상태인데 결제 성공 웹훅이 온 경우 → 자동 환불 처리 + 사용자 알림 (예외 케이스, 아래 참고)
   - 결제 실패 웹훅 → `PAYMENT_FAILED`, 슬롯 즉시 해제

6. **예약 확정 화면 (RSV-03, 프론트)**
   QR 코드 및 예약 정보 표시.

### 반드시 처리해야 할 엣지 케이스

- **슬롯 동시 충돌**: 홀드 생성 단계에서 DB 락/원자적 카운터로 처리, 결제 화면까지 간 뒤
  실패하는 상황이 없도록 선택 시점에 즉시 알럿.
- **결제 성공 + 홀드 만료 레이스 컨디션**: 웹훅 처리 시 슬롯 유효성 재검증 필수.
  만료 후 결제 성공한 경우 자동 환불 + 알림 (정책 담당과 환불 정책 별도 협의 필요).
- **웹훅 중복 수신(멱등성)**: 예약 ID 기준 이미 `CONFIRMED`면 재처리 무시, 이중 QR 발급 방지.
