# [의사결정] 사용자 타입별 로그인 화면 및 관리자 계정 생성 정책

**담당**: 기획 및 서비스 디자인 파트 · **상태**: 초안
---

## 1. 비즈니스 요구사항

### 배경
TravelX의 일반 사용자와 환전소 관리자의 로그인 방식을 결정해야 함. 관리자가 진짜 지점 직원인지 시스템으로 자동 확인하는 기능을 만들지 않기에 우리 시스템이 관리자 계정을 수동 발급해 주기로 계획했으나, 이로 인해 로그인 화면이 각각 분리되는 비효율이 생김. 

관리자도 소셜 로그인으로 통합하면 화면은 하나로 깔끔해지지만, 우리 서비스의 검사 없이 아무나 관리자가 될 수 있는 보안 사고 우려가 있어 이를 막기 위한 별도 승인 로직이나 솔루션이 필요함. 이에 개발 범위 축소와 안전한 관리 사이에서의 타협점을 도출하고자 함.

### 운영 정책

| 항목 | 대안 1: 로그인 화면 분리 및 계정 직접 생성 (기존 안) | 대안 2: 구글 로그인 통합 후 사용자 선택 (대안 안) |
| :--- | :--- | :--- |
| **로그인 화면** | 일반 사용자와 관리자가 접속하는 화면 주소 자체를 다르게 둠 | 모든 사용자가 동일한 구글 로그인 화면 하나만 봄 |
| **계정 생성 방법** | **일반 사용자**: 구글 연동으로 자동 가입<br>**관리자**: 본사가 계약 완료 후 직접 아이디/비밀번호 생성 | **공통**: 구글 로그인 후 최초 1회 본인의 역할(일반 사용자 / 관리자)을 직접 선택함 |
| **진짜 직원인지 확인하는 곳** | 우리 서비스 외부 (우리 서비스 외부에서 검증 과정을 거쳤다고 가정하기에 우리 책임이 없음) | 서비스 내부 (사용자가 선택한 역할에 대해 시스템에서 자동/수동 승인) |

## 2. 의사결정 대안 분석

### 대안 1 — 로그인 화면 분리 및 계정 직접 생성 (기존 안)
* **사용자가 겪는 흐름**: 
  * 일반 사용자: 서비스 메인에서 구글 로그인 버튼을 눌러 바로 가입하고 예약 서비스를 이용함 (→ [TravelX_PRD_v1.3_EN_final.docx](../prd/TravelX_PRD_v1.3_EN_final.docx) §24 참고).
  * 관리자: 파트너 전용 화면에 접속하여 본사에서 메일 등으로 알려준 아이디와 비밀번호를 직접 입력해 로그인함 (→ [admin_descriptions.md](../prd/feedBack/Admin/admin_descriptions.md) ADM-1 참고).

* **서비스적 장점**
  * **초기 개발 범위와 기간 줄이기**: 가입 승인 대기 화면, 지점 매핑 기능, 사업자 확인 화면 등을 설계할 필요가 없어 MVP 출시 일정을 크게 아낄 수 있음.
  * **보안 및 안전성**: 일반 사용자가 관리자 화면으로 잘못 들어오거나 해킹을 시도할 수 있는 통로가 로그인 화면 수준에서 완전히 차단됨.
* **서비스적 단점**
  * **직접 계정을 관리해야 하는 불편함**: 관리자가 비밀번호를 잃어버렸을 때 본사 담당자가 직접 본인 확인 후 비밀번호를 초기화해서 알려주어야 하므로 수동 대응 일이 늘어남.

---

### 대안 2 — 구글 로그인 통합 후 사용자 선택 (대안 안)
* **사용자가 겪는 흐름**: 
  * 로그인 시도: 일반 사용자든 관리자든 똑같이 구글 로그인 버튼 하나만 사용함.
  * 역할 선택: 로그인 완료 후 "관리자로 시작하기" 혹은 "일반 사용자로 시작하기" 중 하나를 골라 진입함.

* **서비스적 장점**
  * **비밀번호 관리 부담 해소**: 관리자도 구글 계정으로 로그인하므로 비밀번호 분실 시 구글에서 비밀번호 찾기를 처리하여 본사가 계정 복구를 도와줄 필요가 없음.
  * **동일한 로그인 화면 제공**: 하나의 로그인 화면만 필요하므로 앱 및 화면 인터페이스 일관성이 향상됨.
* **서비스적 단점**
  * **도용 및 유출 위협**: 별도 확인 절차가 없으면 아무나 "관리자"를 선택해 다른 지점의 환전 금액을 바꾸거나 고객의 예약 신청 목록을 가로채 볼 수 있음.
  * **개발해야 할 기능이 크게 늘어남**: 위 사고를 막기 위해 `환전소 정보 확인`, `본사 승인 화면`, `사용자와 지점 1:1 연결 기능` 등 추가 가입 프로세스를 전부 새로 기획하고 개발해야 하므로 출시일이 지연됨.

## 3. 의사결정 비교

| 비교 항목 | 대안 1 (기존 안 / 추천) | 대안 2 (대안 안) |
| :--- | :--- | :--- |
| **본사 운영팀의 업무량** | 보통 (비밀번호를 수동으로 알려주는 일 발생) | 높음 (새로운 관리자 승인 요청을 늘 확인해야 함) |
| **초기 기획 및 개발 부담** | **매우 낮음** (로그인 페이지를 가볍게 분리하여 구현) | **매우 높음** (직원 등록 및 검토를 위한 관리 화면 필수 추가) |
| **서비스 보안 및 안전성** | **매우 우수** (진입로가 완전히 나뉘어 있어 권한 도용 불가) | 취약함 (잘못 누르거나 가짜 승인 시 지점 정보가 그대로 노출됨) |

---

## 4. 추천하는 방향 및 함께 논의해볼 질문
* **추천 방향**: 초기 MVP 출시 속도와 혹시 모를 정보 유출 문제를 예방하기 위해 **[대안 1: 로그인 화면 분리 및 계정 직접 생성]** 채택을 추천함.
* **함께 논의해볼 질문**:
  * 환전소 계약이 완료된 후, 관리자에게 아이디와 비밀번호를 전달할 때 어떤 보안 방식(예: 계약 이메일 전달 또는 점포 방문 수동 전달)을 사용할 것인지의 운영 규칙.
  * 향후 점포가 너무 많아져 수동 생성이 불가능할 경우를 대비한 자동 승인 도입 로드맵은 어떻게 되는지의 타임라인 설정.
